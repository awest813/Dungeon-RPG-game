## 2024-03-21 - Replace JSON.parse(JSON.stringify) with manual clone
**Learning:** The codebase heavily relied on `JSON.parse(JSON.stringify(...))` for deep copying `Hero` and `Enemy` objects in multiple places (`Game.ts`, `CombatManager.ts`, `DungeonManager.ts`). This is a known performance anti-pattern. Benchmarks showed that a manual deep copy function is ~25x faster (10ms vs 250ms for 100k iterations). `structuredClone` was also tested but proved to be slower than `JSON` in this specific node environment (415ms).
**Action:** When deep copying complex domain objects with known shapes (like `Hero` or `Enemy`), prefer manual spread-based cloning functions over `JSON.parse(JSON.stringify)`. `structuredClone` should also be benchmarked before blindly adopting it, as its performance can vary depending on the environment and object structure.

## 2024-05-18 - Share Babylon.js Materials to reduce Draw Calls and Memory
**Learning:** In the `CombatScene`, creating a new `StandardMaterial` instance dynamically inside a loop for every instantiated entity (e.g., `spawnCombatants` generating unique materials for each hero and enemy) is a severe WebGL anti-pattern. This prevents the GPU from batching the draw calls efficiently, inflating memory usage, and degrading frame rates as the number of entities scales.
**Action:** Define reusable `StandardMaterial` (or other material type) singletons during scene initialization. Apply these shared material instances directly to dynamically spawned meshes instead of instantiating new material objects for each mesh. This simple change reduces the number of generated materials in a standard 4v4 encounter from 8 to 2, drastically cutting down WebGL shader compilation time and draw call overhead.
